# Progress: harness-enforcement-gates

## Original Goal

Investigar y diseñar mecanismos de **enforcement automático** para el plugin
RalphHarness que aborden 5 deficiencias detectadas en el diagnóstico forense de
la spec `mutation-score-ramp` (proyecto ha-ev-trip-planner). El diagnóstico
completo está en `plans/mutation-score-ramp-diagnostic.md`.

Puntos a investigar:

1. **2.3 — Task 1.18 saltado, DEADLOCK no resuelto**: El coordinador saltó una
   tarea `[VERIFY]` obligatoria (1.18) y avanzó a tareas posteriores sin resolver
   el DEADLOCK que el external-reviewer escaló dos veces. Consecuencia material:
   `__init__` quedó en 50.7% vs 51% threshold. Falta un **bloqueo secuencial**
   que impida delegar la tarea N+1 mientras una tarea `[VERIFY]` precedente
   siga en `[ ]`.

2. **3.1 — Fix incompleto de timeout (index_manager.py)**: El coordinador afirmó
   haber corregido un loop infinito, pero `git diff HEAD` mostró CERO cambios en
   el archivo de producción. Pista del usuario: el problema puede ser que
   `git diff` ya no muestre el cambio porque está en un commit o en stage —
   `git diff` a secas puede NO ser suficiente como mecanismo de verificación.
   Investigar qué comparación git es robusta para confirmar que un fix afirmado
   existe realmente en el código (working tree + staged + committed en la rama).

3. **3.3 — Phase A gate NO verde, se procedió a Phase B igualmente**: El
   `design.md` definía que Phase A debe alcanzar gate verde antes de Phase B,
   pero con 2 módulos por debajo del umbral la ejecución avanzó de fase de todos
   modos. Falta un **gate de fase** enforced mecánicamente por el coordinador.

4. **4.2 — `.metrics.jsonl` vacío**: El archivo de métricas existía pero quedó
   con una sola línea vacía; no se registró ninguna métrica pese a que la spec
   trataba precisamente sobre métricas de mutation testing.

5. **4.3 — `_unmark_task.py` vacío**: El script placeholder existía pero nunca
   se implementó. Relacionado con la deficiencia 2.2 (desmarcado manual de
   tareas verificadas). Debería validar que una tarea no tiene entrada PASS en
   `task_review.md` antes de permitir desmarcarla.

Entregable de esta fase: un **plan de investigación** que analice cada punto,
su causa raíz en el plugin, y las opciones de enforcement viables.

## Learnings

- Master root cause for ALL 5 points: "prose-as-enforcement" — rules live as
  English in agent/reference markdown where the LLM can skip them, not as
  deterministic shell gates in hooks.
- The coordinator's ONLY advancement gate is `taskIndex < totalTasks`
  (`stop-watcher.sh:676`). No sequential `[VERIFY]` gate, no phase-exit gate.
- `implement.md:290-292` explicitly TOLERATES task skipping ("No correction:
  allow execution to continue") when `taskIndex > completed`.
- `signals.jsonl` is mechanically enforced (HOLD-GATE); `chat.md` is NOT. The
  §2.3 DEADLOCK landed in `chat.md` so no mechanical gate fired. Fix: route
  reviewer DEADLOCK to `signals.jsonl`.
- `_unmark_task.py` is NOT plugin code — zero hits in `plugins/ralphharness/`.
  It was a spec-local artifact. "Implementing it" = designing an integrity
  guard the plugin currently lacks.
- `write-metric.sh` is production-ready and flock-safe; metrics are empty only
  because the ONLY caller is an LLM-discretionary coordinator prompt block
  (`implement.md:680-708`). Fix: move the call into `stop-watcher.sh`.
- `git diff HEAD` (working-tree) and `git diff HEAD~1` (single commit) each
  cover only some states; robust fix-presence = `git merge-base` three-state
  diff (committed ∪ staged ∪ working-tree).
- Reusable machinery already exists: PRE-EXEC-GATE pattern, `flock` fds
  (200/201/202), HOLD-GATE, circuit breaker. The fix is to WIRE it, not invent.
- `stop-watcher.sh` is append-only by policy (loop-safety.md Decision 3).
- Coordinator auto-recovery (diagnostic §3.2) is NOT one of the 5 points and
  conflicts with circuit-breaker "manual reset only" — recommend out of scope.

## Interview Responses

### Requirements Interview (from requirements.md)
- Point 1 (VERIFY gate): Block ALL preceding unsatisfied `[VERIFY]` tasks — no
  deferral marker. The plugin has no notion of deferral; keep it simple/strong.
- Point 3 (Phase gate): Synthetic `[VERIFY] Phase X exit gate` task emitted by
  task-planner as the last task of each phase. Reuses Point 1's gate; no
  coupling to metric semantics.
- Point 5 (Un-mark guard): Tiered escalation. On illegitimate `[x]`→`[ ]`
  detection: (1) emit DEADLOCK to HALT advancement; (2) BEFORE escalating to a
  human, run a full subagent review/triage to filter false positives — if the
  subagent understands the problem and can resolve it, resolve it; (3) only
  escalate to the human when it is a genuine conflict needing human
  intervention. No blind auto-revert.
- Point 4 §3.2 coordinator auto-recovery: OUT of scope. Conflicts with
  circuit-breaker "manual reset only"; deserves its own spec.
- Chosen approach: Full feature set — all 5 enforcement gates, each as a
  deterministic shell gate (NFR enforcement-as-shell).

### Design Interview (from design.md)
- Mark snapshot persistence: `.ralph-state.json` field (e.g. `taskMarkSnapshot`),
  not a sidecar file — single source of truth, follows engine-state-hardening
  conventions, atomic with other state.
- Tier 2 integrity triage agent: use the `bmad-consensus-party` skill if
  available (exact skill name confirmed present at
  `.claude/skills/bmad-consensus-party/`). It runs BMAD Party Mode iteratively
  until agents reach consensus. Fallback when the skill is absent: invoke 2–3
  subagents (e.g. external-reviewer + qa-engineer) that perform a consensus-
  seeking review of the un-mark.
- Exit-gate naming: task-planner ALWAYS appends a distinct
  `[VERIFY] Phase X exit gate` task, even when the phase already ends with a
  `[VERIFY]` checkpoint — unambiguous mechanical marker.
- Chosen approach: Append-only shell gates extending existing infrastructure
  (`stop-watcher.sh` functions + one helper script), no new control flow.
