# Progress: reviewer-warmup

## Original Goal

Mejorar el desempeño de ejecución del repositorio Smart Ralph, cubriendo las
deficiencias detectadas en un análisis forense de 6 specs (~200 tareas) ejecutadas
la última semana (context-middleware, pre-execution-critic, pair-debug-auto-trigger,
collaboration-resolution, signal-log-and-ci-autodetect).

Hallazgo principal — **cold-start del external-reviewer**: el external-reviewer
arranca con `chat.md` vacío y necesita múltiples ciclos para acumular contexto.
En context-middleware esto produjo una fase de "struggle" de 8 ciclos (~43 min,
WARNING→FAIL→INTENT-FAIL→DEADLOCK) antes de entrar en una fase de "momentum" donde
revisó 24+ tareas sin error y capturó bugs reales. La hipótesis del usuario —
"a los agentes les cuesta leer chat.md, cuando cogen carrerilla leen bien" —
quedó validada. Ninguna spec del epic engine-roadmap abordó el warm-up del reviewer;
agent-chat-protocol definió el formato de chat.md pero no este problema.

Deficiencias a cubrir:
1. **Cold-start del external-reviewer** — empieza sin contexto; 30-40 min de overhead
   al inicio de cada spec.
2. **Executor lee design docs sin output visible** — el reviewer no puede distinguir
   "processing normal" de "stagnation", causando DEADLOCK innecesario.
3. **chat.md crece sin límite** — 966 líneas en context-middleware; lectura incremental
   desde lastReadLine es lenta.
4. **No hay sync de contexto para el reviewer** — a diferencia del executor que tiene
   .progress.md como fuente de verdad, el reviewer solo depende de chat.md vacío.

Próximos pasos propuestos por el usuario (a validar en research):
- Bootstrap phase: el external-reviewer lee chat.md completo al inicio, o un contexto
  inicial en task_review.md con el estado del spec.
- Signal "READING": el executor escribe un signal cuando está leyendo/entendiendo,
  para que el reviewer no lo confunda con stagnation.
- chat.md trimming: el coordinator compacta chat.md antes de que crezca demasiado.

## Learnings

- Deficiency #3 (chat.md unbounded growth) is ALREADY SOLVED — context-middleware shipped `hooks/scripts/condense-context.sh` + `lib-context.sh` (proactive/reactive/emergency condensation, archives, AC-1.1–AC-1.8). reviewer-warmup must NOT reimplement trimming; only compose with it. Solution 3 is out of scope.
- `signals.jsonl` enum is OPEN — `append_signal()` (lib-signals.sh:15) validates only that the line is valid JSON, not the `signal` value. A new signal kind needs no schema migration.
- `active_signal_count()` (lib-signals.sh:25-30) only counts blocking signals `HOLD|PENDING|URGENT|DEADLOCK`. A new non-blocking heartbeat/`READING` kind is ignored by the HOLD gate automatically — no coordinator change needed to keep it non-blocking.
- `chat.md` legend already defines `ALIVE` (heartbeat) and `STILL` (alive, no progress) collaboration markers, treated by coordinator as "ignore, do not block" (coordinator-pattern.md:246). A liveness signal partially exists conceptually but is never EMITTED by the executor during read phases nor CONSUMED by the reviewer to suppress a stagnation verdict.
- external-reviewer Section 0 bootstrap (lines 19-33) sets `chat.reviewer.lastReadLine` to the FULL current line count — it deliberately SKIPS all existing chat.md history. Root cause of cold-start.
- The reviewer's "progress-real" stagnation check (Section 3b Step 6) exists ONLY for VE/E2E mid-flight tasks. Non-VE tasks (like the design-doc-reading task 1.3) have NO liveness check — silence is misread as stagnation. This caused the unnecessary DEADLOCK.
- Prior art: OpenAI Codex Issue #16900 documents the identical "long-running healthy subagent treated as stalled" problem — validates modeling liveness as a freshness-bearing heartbeat, not a one-shot flag.
- Recommended scope: (1) executor liveness heartbeat to signals.jsonl (highest value — prevents the false DEADLOCK), (2) reviewer bootstrap full-read of chat.md+.progress.md+git, (3) doc updates. NOT chat.md trimming.
