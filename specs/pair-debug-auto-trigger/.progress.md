## Design Phase — Rewrite for Expanded Scope (2026-05-16)

Requirements were rewritten to add US-6, US-7, FR-14–FR-20. design.md fully rewritten.

**New scope learnings**:
- Two-instance dimension is orthogonal to the trigger. The trigger (3 mechanical conditions) is unchanged; what changed is WHERE the Driver/Navigator roles execute — same instance, second Claude Code instance, or a foreign runtime.
- Filesystem-only coordination is load-bearing for cross-provider pairing: a Task-tool handoff would tie both roles to one Claude Code process and make foreign runtimes impossible. chat.md + signals.jsonl + .ralph-state.json (read-only for role files) are the only coordination channel.
- The two role files (`pair-debug-driver.md`, `pair-debug-navigator.md`) are prompt markdown, not executable code — they package EXISTING spec-executor / external-reviewer behavior for standalone invocation. They register NO new `subagent_type`. Consequence for tests: they are verified structurally (grep for Section 0 Bootstrap, anti-anchoring rule, no plugin-only path) and behaviorally only at e2e level.
- Section 0 Bootstrap (from external-reviewer.md:19-33) is the exact self-discovery pattern both role files copy — read specs/.current-spec, set basePath, confirm phase, honor chat.md signals, announce, begin loop.
- Known BROKEN pattern to NOT reproduce: implement.md "Parallel Reviewer Onboarding" (lines ~312-325) tells the user `@external-reviewer` with no file path and no copy step; the `.roo/`/`.qwen/` external-reviewer files were hand-created with no generator. - Export must be idempotent/non-destructive: existing destination file → overwrite/skip prompt, never silent clobber.
- implement.md is now a FOURTH append-only edit (placement step after Parallel Reviewer Onboarding, before Coordinator Prompt). collaboration-resolution.md remains the only one-value change (line 53, already applied in requirements phase).
