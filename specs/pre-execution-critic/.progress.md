- OpenHands SDK security system is a multi-layer pipeline: ActionEvent -> SecurityAnalyzer -> SecurityRisk -> ConfirmationPolicy -> Allow/Block
- Four SecurityRisk levels: LOW (read-only), MEDIUM (modify user data), HIGH (dangerous ops), UNKNOWN (not analyzed)
- Three confirmation policies: AlwaysConfirm, NeverConfirm, ConfirmRisky (default threshold=HIGH, configurable confirm_unknown)
- Hooks use exit codes: 0=allow, 2=block, other non-zero=non-blocking error. ONLY exit code 2 blocks.
- PreToolUse hook runs BEFORE tool execution and CAN block (exit code 2)
- PatternSecurityAnalyzer catches known threats via regex (rm -rf, eval, curl|sh)
- PolicyRailSecurityAnalyzer catches composed threats (pip | exec)
- EnsembleSecurityAnalyzer takes worst-case across all child analyzers
- Defense-in-depth: pattern + policy rails + LLM analyzer = speed + breadth
- "No hard-deny at analyzer boundary" confirmed that HOOKS are the enforcement mechanism
- Known limitations: execute_tool() bypasses checks, thinking_blocks not scanned
- Custom SecurityAnalyzer: inherit SecurityAnalyzerBase, implement security_risk()
- Configurable Security Policy: Jinja2 template rendered into system prompt

## Learnings (Requirements Phase)

- Max-severity combiner is the load-bearing rule — when multiple layers fire, highest risk wins. Layer 1 (role-contract) is the only hard-block; Layer 2 (shell-pattern) only escalates to HIGH which routes through ConfirmRisky.
- Scope decision: only ConfirmRisky(threshold=HIGH, confirm_unknown=true) ships — no policy selection. Keeps the gate deterministic and avoids config surface.
- Fail-safe principle: indeterminate state (missing role-contracts.md, unknown role, script error, other non-zero exit) must route to UNKNOWN→confirmation, never fail-open to allow. NFR-3/NFR-4 capture this.
- Dependency on Spec 3: Layer 1 parses the `## Access Matrix` markdown table in role-contracts.md. If Spec 3 ever changes that table shape, the Layer 1 parser breaks — flagged in Verification Contract dependency map and FR-2/FR-11.
- Open ambiguity (design must resolve): tasks.md tasks are prose, but the check needs structured inputs (intended write paths + intended shell command). The coordinator must derive these before delegation — no obvious existing mechanism. Logged as an Unresolved Question.
- Reuses Spec 4 infra fully: `append_signal` (flock fd 202, validates JSON before write) and the immutable append-only log. New event type `security-decision` co-exists with control signals — no new lock, no new file.

### Design Interview (from design.md)
- Task → structured input: coordinator parses the **Files:** metadata field already present in every task block (templates/tasks.md confirms it); shell command input comes from the **Verify:** field (no separate **Commands:** field exists). Tasks missing **Files:** default to UNKNOWN→confirm. Deterministic, no prose guessing.
- Confirmation capture: inline coordinator prompt — on a confirm verdict the coordinator pauses, asks the human inline, and on approval appends the follow-up security-decision event then dispatches. Synchronous; matches existing HOLD-gate UX. No PENDING/ACK signal round-trip.
- Schema location: extend the existing schemas/spec.schema.json with the security-decision event shape (no new signals.schema.json — control events aren't schema-formalized today, keep one schema file).
- Chosen approach: extend existing infrastructure — new pre-execution-check.sh script + lib-precheck.sh-style helpers if needed, gate inserted in implement.md immediately before the existing HOLD-GATE, reuse lib-signals.sh append_signal for the audit event.

## Learnings (Design Phase)

- Layer 1 short-circuits the pipeline: a role-contract violation produces a hard-block BEFORE the max-severity combiner runs. The combiner only sees Layer 1's non-violation outputs (clear/UNKNOWN). This keeps AC-1.3 (hard-blocks not subject to ConfirmRisky) structurally guaranteed, not just policy-enforced.
- UNKNOWN must rank ABOVE HIGH in the severity order. Both route to confirm, but the recorded `risk` field must reflect indeterminacy for accurate replay — a UNKNOWN masked as HIGH would mislead incident review.
- Layer 1 glob-vs-glob: task `**Files:**` carries globs (`src/pages/*.tsx`) AND the Access Matrix Writes/Denylist cells are also patterns. Matching is bash `[[ path == pattern ]]` with extglob; no-Writes-match = out-of-bounds (fail-safe, never fail-allow).
